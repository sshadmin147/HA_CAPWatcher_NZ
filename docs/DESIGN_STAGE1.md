## Problem Statement

Home Assistant users in New Zealand (and eventually any region) have no integrated way to display Common Alerting Protocol (CAP) weather and civil defence alerts on their dashboards. Weather alerts are critical safety information, but integrating them requires manual setup of multiple feeds, inconsistent data sources, and no native card designed for CAP data.

The opportunity: build a HACS addon that ingests CAP feeds (MetService, NEMA, GeoNet, Auckland Emergency Management) and presents them as native Home Assistant entities and a standards-compliant card that users can add to their dashboards.

## What Makes This Cool

**Compliance as a feature, not a burden.** Most alert integrations display alerts "nice-looking" and hope for the best. This one is built from day one around NZ-CAP and MetService standards — same color codes, same severity hierarchy, same mandatory fields as official government sources. When someone sees an alert on their HA dashboard, it looks and behaves exactly like alerts they see on gov websites and their phone.

**Multi-feed, multi-region from day one.** Users aren't locked into one region. Monitor home (Auckland), family (Wellington), parents (Southland) — each feed is independent, each can be dismissed or kept visible. The architecture scales to global CAP feeds, not just NZ.

**Built for both dashboards and automations.** The card is beautiful (the first priority). But the underlying entity structure exposes everything — severity, urgency, certainty, geometry — so automation nerds can trigger actions (notifications, sirens, mode switches) based on alert presence/severity.

## Constraints

- **Standards-first:** If NZ-CAP says it's mandatory, it ships. No compromises on data fidelity.
- **Safe, vetted feeds:** Default feeds shipped in integration's config file (based on `.information/cap-feeds.md`): official NZ, by-region, with/without earthquakes, Auckland enhanced. Users can enable/disable defaults and add custom feeds via config editor. User selects which to enable in config flow.
- **Single integration, multiple devices:** One HACS install, user configures which feeds to monitor, one device per feed.
- **Polling strategy:** 1-4 times per minute (user-configurable presets: 15s, 30s, 45s, 1m, 2m, 5m). Enforce per-feed rate limits (20 req/min per IP, server caches 60s). Use request queuing/serialization across all feeds to stay within limit — polls are queued and executed sequentially across feeds, not in parallel.
- **Entity naming:** Use CAP alert's UUID as entity suffix (first 8 chars for readability; HA entity IDs max 255 chars). Example: `sensor.cap_alerts_nz_auckland_a1b2c3d4`. This prevents ID collisions even if feed provides duplicate IDs.
- **Dual color scheme:** Severity maps to NZ-CAP standard colors; stored as normalized severity string, not hex.
- **Future-proof for NZAlerts migration:** When this repo moves under the NZAlerts brand, the integration is already compliant and maintainable.

## Premises

1. **Mandatory fields are always enabled.** If the NZ-CAP standard says a field is mandatory, it appears in the entity. No opt-out in the base product.
2. **Multiple feeds simultaneously.** Users can monitor multiple feeds at once (different regions, different sources). Each feed = one device + one poller.
3. **One entity per alert.** Each active alert from any feed becomes its own Home Assistant sensor entity, grouped under a device.
4. **Severity strings, not hex codes.** Entities store normalized severity ('extreme', 'severe', 'warning', 'watch', 'info'). Colors are applied by the card/CSS, not stored in data.
5. **Configurable polling with presets.** Users choose polling interval from: 15s, 30s, 45s, 1m, 2m, 5m. Defaults to 1m (matches server cache).

## Approaches Considered

### Approach A: Lightweight Polling Service
- One simple polling task per feed, manual entity lifecycle
- Entities created on startup, updated on poll, deleted on expiry
- Less boilerplate, ships fast
- **Tradeoff:** No built-in retry logic, backoff, or error resilience. Harder to extend later. If network glitches, alerts might stall until manual restart.
- Effort: Small | Risk: Low (initially) → Medium (as bugs appear) | Maintainability: Medium

### Approach B: HA Coordinator Pattern
- Each feed = one async Coordinator instance
- Coordinator handles polling, automatic backoff, retries on failure
- Follows Home Assistant conventions — easier for other developers to understand
- Entity lifecycle is automatic (created/deleted as alerts change)
- **Tradeoff:** More boilerplate upfront, but scales well and follows HA idioms
- Effort: Medium | Risk: Low | Maintainability: High

### Approach C: Coordinator + Helper Entities (Recommended)
- Approach B + template entities for aggregated state
- In addition to raw per-alert sensors, create helper sensors for:
  - `alert_count`: Number of active alerts for this feed
  - `highest_severity`: Highest severity among active alerts
  - `latest_headline`: Most recently issued alert headline
- Gives both dashboard users (raw alerts) AND automation nerds (aggregated state)
- **Tradeoff:** ~1 extra hour of work, but serves full audience immediately
- Effort: Medium | Risk: Low | Maintainability: High

## Recommended Approach

**Approach C: Coordinator + Helper Entities**

**Why:** Approach A ships fast but leaves technical debt. Approach B is solid but doesn't serve the automation-focused users (the "nerds") on day one. Approach C costs only 1 extra hour and immediately unlocks both use cases.

The Coordinator pattern is Home Assistant's standard for polling integrations — your code becomes maintainable by the broader HA community. The helper entities are lightweight template sensors that aggregate the raw alert data, so automations can trigger on "any alert active" or "extreme alert detected" without parsing individual sensors.

**Architecture Overview:**

```
HA Integration (hacs addon)
├── Config Flow
│   └── User selects feeds to monitor (checkboxes: "Auckland Enhanced", "Wellington", etc.)
│
├── One Coordinator per Feed
│   ├── Polling task (async, respects user's interval)
│   ├── CAP XML parser (normalizes severity → 'extreme'/'severe'/etc.)
│   ├── Entity lifecycle management
│   └── Error handling + backoff
│
├── Device: cap_alerts_nz_[feed_name_slug]
│   ├── sensor.cap_alerts_nz_[feed]_[alert_uuid_short]
│   │   ├── state: "Wind Warning"
│   │   ├── severity: "moderate"
│   │   ├── certainty: "observed"
│   │   ├── issued: "2024-06-26T10:00:00Z"
│   │   ├── onset, expires, area, description, instructions
│   │   ├── geometry_polygon: "<polygon coordinates>" (separate attribute to avoid bloat)
│   │   └── (all mandatory NZ-CAP fields)
│   ├── sensor.cap_alerts_nz_[feed]_[alert_uuid_short]
│   └── [more alerts with unique UUIDs]
│
└── Helper Entities (per device)
    ├── template.cap_alerts_[feed]_count → "2"
    ├── template.cap_alerts_[feed]_highest_severity → "moderate"
    └── template.cap_alerts_[feed]_latest_headline → "Wind Warning"
```

**Stage 1 Deliverable:** Entity layer + raw sensors + helper aggregates (ready for card to consume).

**Stage 2 Deliverable:** Custom HA Lovelace card that:
- Renders severity → NZ-CAP colors (via CSS variables, as in web UI)
- Shows geometry (mini-map of affected area)
- Expandable details (description, instructions, source)
- "No active alerts" state when empty

## Error Handling & Resilience

**Coordinator error strategy:**
- **Timeout/unreachable feed:** Log warning, retry with exponential backoff (5s, 10s, 30s, 2m, 5m max). After 5 failed attempts, create a diagnostic entity `binary_sensor.cap_alerts_[feed]_error` (on=feed offline). Remove error entity when feed recovers.
- **Malformed XML:** Log error, skip alert, continue parsing remaining alerts.
- **Missing mandatory field:** CAP feeds MUST include severity. If missing, fail loudly and skip alert. If severity is missing from NZAlerts feed, it's a data quality issue on NZAlerts' side to fix, not the integration's responsibility to work around.
- **Rate limited (HTTP 429):** Respect `Retry-After` header, back off to 5-minute polling temporarily.

**State persistence on HA restart:**
- Coordinator stores last successful poll in HA's restore service. On restart, re-create alert entities from last known state. On next successful poll, update all entities. This ensures alerts don't disappear during restarts.

## Open Questions

1. **Geometry rendering:** Should the card include an inline SVG mini-map of the affected area (polygon from CAP geometry field), or just text? Adds visual richness but complexity. (Stage 2)
2. **Card config:** Should the card be configured to a specific device (one card per feed), or can one card display multiple feeds? (Stage 2)
3. **Dismissible alerts:** Should users be able to dismiss individual alerts in the card (browser-local storage), or always show all active alerts? (Stage 2)
4. **Severity enum mapping:** Confirm fallback for unknown severity values (proposed: map to 'info'). (Stage 1, clarification)
5. **Feed credentials:** Do any feeds require API keys? If so, how are they stored in HA config? (Stage 1, clarification)

Items 1-3 don't block Stage 1, but should be settled before Stage 2 (card). Items 4-5 are clarifications needed for Stage 1 implementation.

## Success Criteria

- Entity layer ingests CAP feeds without losing mandatory fields
- All normalized severity values match NZ-CAP standard
- Multiple feeds run independently (no crosstalk, no cascading failures)
- Polling respects user's interval choice + rate limits
- Helper entities correctly aggregate alert state
- Integration handles stale data gracefully (old alerts auto-delete when expired)
- Configuration is user-friendly (config flow, not YAML)

## Next Steps

2. **Build CAP parser** — Python to normalize CAP XML → entity attributes. Use CAP alert UUID for stable entity IDs. Handle malformed/missing-field gracefully.
3. **Set up Coordinator scaffold** — HA integration boilerplate + polling loop + error recovery (exponential backoff, diagnostic entities).
4. **Implement entity lifecycle** — create/update/delete sensors as alerts change. Auto-cleanup expired alerts on each poll.
5. **Create feeds config file** — YAML file bundled with integration defining default NZAlerts ATOM feed URLs (feed name, endpoint URL, polling interval presets). Ship with defaults from `.information/cap-feeds.md`. Load at integration startup (HA best practice: once at startup, reload on HA restart or manual reload).
6. **Wire config flow** — let users select feeds from config file (checkboxes: Auckland Enhanced, Wellington, All NZ, etc.) and option to add custom feeds. Custom feed validation: fetch the URL, parse as Atom XML, extract first CAP entry, validate it has required fields (severity, urgency, certainty). If parsing succeeds, accept the feed; if it fails, show error "Invalid CAP feed URL."
7. **Add CAP validator** — Custom feed validation happens in the Coordinator on first poll (lazy validation). If a feed URL is unreachable or returns invalid XML, the error handling creates a diagnostic entity (same as other feed errors).
7. **Add helper entities** — template sensors for alert_count, highest_severity, latest_headline (computed from raw alerts, no polling).
8. **Test multi-feed setup** — ensure two pollers run independently without interference. Test rate limit enforcement.
9. **State persistence testing** — verify alerts survive HA restart (restore service integration).
10. **Manual QA** — ingest real CAP data, verify entity structure and severity mapping. Test severity inference for edge cases.
11. **Document** — README for users (how to install, configure, manage multiple feeds, use on dashboard).
