## Config Flow Tests

**Custom Feed Validation**
- Valid Atom feed URL → accepted, stored in config
- Invalid URL (404) → error message shown, feed not added
- Malformed XML response → error message shown, feed not added
- Valid XML but missing CAP fields → error message shown, feed not added
- Valid CAP feed → accepted, ready for polling

## Coordinator Tests (Per Feed)

**Happy Path Polling**
- Fetch feed → parse Atom → extract alerts → create sensors
- Alert with all mandatory fields → sensor created with correct state/attributes
- Update existing alert → sensor attributes updated, state refreshed
- Alert expires → sensor deleted automatically

**Error Handling & Recovery**
- Feed timeout → retry with exponential backoff (5s, 10s, 30s, 2m, 5m)
- HTTP 429 (rate limited) → respect Retry-After header, backoff to 5m polling
- Malformed XML → log error, skip alert, continue parsing remaining alerts
- 5 consecutive failures → create `binary_sensor.cap_alerts_[feed]_error` (on)
- Feed recovers after error → remove error entity, resume normal polling

**Severity Inference**
- CAP severity field present → use it directly
- Severity missing but urgency/certainty available → infer severity correctly
- Multiple fields missing → cannot infer → skip alert, log error

**Entity Lifecycle**
- First poll: create sensors for all active alerts
- Second poll (no change): sensors unchanged
- Third poll (one alert expired): delete expired sensor, keep others
- Entity ID uses UUID (first 8 chars) → stable across restarts

**State Persistence**
- HA shutdown → Coordinator stores last-known alert state in restore service
- HA restart → alert sensors recreated from stored state
- First poll after restart → all alerts updated with current data

## Helper Entity Tests

**Aggregated State Computation**
- Zero active alerts → `alert_count="0"`, `highest_severity="none"`, `latest_headline=""`
- One alert (moderate) → `alert_count="1"`, `highest_severity="moderate"`, `latest_headline="[headline]"`
- Multiple alerts (moderate, severe, warning) → `alert_count="3"`, `highest_severity="severe"` (highest wins), `latest_headline="[most recent]"`
- Alert expires → helpers recalculated, count decreases, severity updates if needed

## Multi-Feed Tests

**Feed Isolation**
- Feed A enabled, Feed B enabled → both polling independently
- Feed A timeout → Feed B continues normal polling
- Delete Feed A → Feed B unaffected, keeps polling
- Error on Feed A (5 failures) → Feed A error entity created, Feed B still polling
- Feed A recovers → Feed A error entity removed, Feed A resumes polling

## Integration Tests

**Config Change Scenarios**
- User adds Feed B while Feed A polling → Feed B polling starts, Feed A unaffected
- User disables Feed A → Feed A polling stops, entities cleaned up, Feed B continues
- User changes polling interval on Feed A → next poll respects new interval

## Edge Cases

**Alert Data Edge Cases**
- Alert with maximum-length description (>500 chars) → handled correctly
- Alert with special characters in headline → entity state stores correctly
- Alert with no geometry polygon → handled gracefully (attribute missing OK)
- Duplicate alert IDs in same feed → UUID collision handling (shouldn't happen, but fail gracefully)

**Timing Edge Cases**
- Alert expires mid-poll → handled in next poll cycle
- HA restart during poll → poll interrupted, state restored on next cycle
- User manually reloads integration → coordinator resets, re-polls all feeds

## Test Framework & Tools

- **Unit tests:** pytest (standard for HA Python integrations)
- **Mocking:** `unittest.mock` for HTTP calls, feed responses
- **Integration tests:** HA test helper fixtures (HA provides mock Home Assistant instance)
- **Test data:** Use real CAP XML samples from `.information/cap-feeds.md` feed examples
